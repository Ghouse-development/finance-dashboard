# 入金管理モジュール｜画面設計書
作成日：2026/03/17
ステータス：設計中（財務チーム確認待ち事項あり）
対象：G-Force 連携予定モジュール（当面は単独アプリとして先行構築）

---

## 1. 概要

### 目的
施主からの入金（5回払い）を物件ごとに管理し、入金確認→チャット通知までを一元化する。
手作業による入金確認・転記作業をなくし、漏れ・遅延を防ぐ。

### 対象ユーザー
| 役割 | 主な操作 |
|------|---------|
| 購買・発注担当 | 入金予定マスターの登録・更新 |
| 財務担当 | 入金チェック（消込）・差異確認 |
| 管理職・経営層 | ダッシュボードで全体把握 |

### 技術スタック
- フロントエンド：Next.js（G-Force と同一基盤）
- データベース：Supabase（G-Force の本番DBへのテーブル追加 or 独立DB ※上席B確認済み）
- 通知：Google Chat Webhook（通知先は財務チーム確認後に設定）
- デプロイ：GitHub → Vercel

### 画面1：ダッシュボード（`/nyukin`）

#### 目的
全物件の入金状況を一目で把握する。未入金・遅延のアラートを即確認できる。

#### KPIカード（上部4枚）
| カード名 | 表示内容 | 備考 |
|----------|---------|------|
| 今月の入金予定（合計） | 対象月の入金予定額合計・対象物件数 | 対象月フィルターと連動 |
| 入金済み | 入金済み合計額・消化率（%） | — |
| 未入金（今月期限） | 予定日超過または今月期限の未入金合計・件数 | 要確認件数をバッジ表示 |
| 来月以降の予定 | 翌月以降の入金予定合計 | ※承認フロー項目は財務確認後に設定 |

#### テーブル：直近の入金状況
| 列名 | 内容 |
|------|------|
| 物件名 | 物件名（クリックで物件別詳細へ遷移） |
| 入金区分 | 申込金／契約金／着工金／中間金／引渡金 |
| 予定日 | 入金予定日 |
| 金額 | 入金予定額（税込） |
| 状態 | 入金済 / 未入金 / 予定（バッジ表示） |

#### テーブル：物件別入金進捗
| 列名 | 内容 |
|------|------|
| 物件名 | — |
| 契約金額 | 施主との契約金額（税込） |
| 入金済 | 入金済み累計額 |
| 進捗 | プログレスバー（入金済 / 契約金額） |

- [ ] 「来月以降の予定」カード：承認フローのステップ数・表示方法

### 画面2：入金予定マスター（`/nyukin/master`）

#### 目的
物件ごとに5回分の入金スケジュールを登録・管理する。

#### フィルター
- 対象月（年月選択）
- 物件（全物件 or 物件名で絞り込み）

#### テーブル
| 列名 | 内容 | 必須 |
|------|------|------|
| 契約番号 | G-Force の契約番号（連携キー） | ○ |
| 物件名 | 物件名 | ○ |
| 施主名 | 施主名 | ○ |
| 入金区分 | 申込金／契約金／着工金／中間金／引渡金 | ○ |
| 入金予定日 | 入金を受け取る予定日 | ○ |
| 入金予定額（税込） | 金額 | ○ |
| 入金割合（%） | 契約金額に対する割合（自動計算） | 自動 |
| 備考 | 特記事項 | 任意 |

#### 操作
- 新規登録ボタン：モーダルまたは別画面で1レコード追加
- 行クリック：編集モーダルを開く
- 削除：確認ダイアログ付き

### 画面3：入金チェック（`/nyukin/check`）

#### 目的
入金確認業務のメイン画面。チェックボックスで消込し、自動通知を送る。

#### フィルター
- 対象月（年月選択）

#### テーブル
| 列名 | 内容 | 備考 |
|------|------|------|
| 消込チェック | ☑ / □ | チェック時に通知送信 |
| 物件名 | — | クリックで物件別詳細へ |
| 施主名 | — | — |
| 入金区分 | — | — |
| 予定日 | 入金予定日 | 期限超過は色で警告 |
| 予定額 | — | — |
| 実入金日 | 実際に入金された日（手入力） | ※将来的に銀行API照合 |
| 実入金額 | 実際の入金額（手入力） | — |
| 差異 | 予定額 − 実入金額 | 自動計算。±0以外は警告表示 |
| 通知 | 送信済 / 未送信 | — |

#### チェック時の動作（確定済み）
1. ☑ をオンにする
2. 実入金日・実入金額を入力（未入力の場合は確認ダイアログ）
3. Google Chat Webhook で担当者に通知送信
4. 通知列が「送信済」に変わる

- [ ] 通知先のチャットスペース・Webhook URL（財務チームへ確認中）
- [ ] 銀行口座の一覧と照合方法（将来的に銀行IB CSV と突合する想定）

#### 上部：物件概要カード
| 項目 | 内容 | 備考 |
|------|------|------|
| 物件名 | — | — |
| 施主名 | — | — |
| 契約番号 | G-Force の契約番号 | G-Force 連携後に自動取得 |
| 契約金額（税込） | — | — |
| 着工日 | — | — |
| 引渡予定日 | — | — |
| 担当営業 | — | G-Force 連携後に自動取得（現在はダミー） |

#### 中段：入金サマリーカード
| 項目 | 内容 |
|------|------|
| 入金済み（合計） | — |

#### 下部：タイムライン

- 状態：入金済（緑）/ 未入金（黄）/ 予定（青）

## 4. データ設計（Supabase テーブル案）

### `payment_schedules`（入金予定マスター）
| カラム名 | 型 | 説明 |
|---------|-----|------|
| id | uuid | PK |
| property_id | uuid | 物件ID（G-Force の properties テーブルと連携） |
| round | integer | 入金回次（1〜5） |
| category | text | 入金区分（申込金／契約金／着工金／中間金／引渡金） |
| scheduled_date | date | 入金予定日 |
| scheduled_amount | integer | 入金予定額（税込、円） |
| created_at | timestamptz | — |
| updated_at | timestamptz | — |

### `payment_records`（入金実績・消込）
| カラム名 | 型 | 説明 |
|---------|-----|------|
| id | uuid | PK |
| schedule_id | uuid | FK → payment_schedules.id |
| actual_date | date | 実入金日 |
| actual_amount | integer | 実入金額（税込、円） |
| checked_by | uuid | 消込した担当者（Supabase Auth のユーザーID） |
| checked_at | timestamptz | 消込日時 |
| notified | boolean | 通知送信済みフラグ |
| notes | text | 備考 |

### 連携ポイント
- `property_id` は G-Force の `properties` テーブルの ID を参照
- G-Force 連携前は入金管理アプリ内に簡易的な物件マスターを持つ（後で差し替え）

## 6. G-Force 組み込み時の対応方針

上席B確認済み：**まず単独モジュールとして先行構築し、後で G-Force に繋ぎ込む**

### 繋ぎ込み時の作業
1. `property_id` の参照先を入金管理の簡易マスター → G-Force の `properties` テーブルに切り替え
2. 担当営業・施主名などを G-Force から自動取得するよう更新
3. G-Force のナビゲーションに「入金管理」メニューを追加
4. Supabase のテーブルを G-Force の本番DB に移行（上席B と共同作業）
